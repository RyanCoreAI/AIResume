# 安全与隐私说明

简喵处理简历、目标岗位、头像、投递文案和导出 PDF，属于敏感个人数据。本页说明当前产品设计中的主要安全边界，以及用户仍然需要注意的风险。

## 攻击面与防护

| 攻击面 | 已有防护方向 | 真实风险 |
|---|---|---|
| 简历富文本 XSS | 前端 DOMPurify；后端保存草稿、版本、分享快照、AI 改写落库前再次净化 | 仍需持续测试所有 `v-html`、富文本编辑器和 AI 输出入口 |
| 越权访问 | 简历、版本、岗位会话、PDF 任务、分享快照等按用户归属或 token 校验 | 需要持续补 IDOR 回归测试 |
| 登录态 | JWT 放 HttpOnly Cookie；CSRF 双提交 token；CORS 白名单 | 生产域名、Cookie Secure / SameSite、Cloudflare 代理配置必须保持一致 |
| 头像上传 | 文件大小限制、Content-Type 与 magic bytes 校验、重新编码、随机文件名 | 头像 URL 可能是公开资源，不应上传敏感图片 |
| 文件导入 | 类型、大小、文件签名、压缩包安全检查 | 文档解析始终是高风险入口，需要限制格式和大小 |
| PDF 导出 | 后端生成内部预览 URL；Puppeteer host 白名单；阻断危险协议和 metadata IP | PDF 渲染链路复杂，必须保留失败日志和浏览器导出兜底 |
| 分享链接 | 长随机 token；短链接只是展示层；支持撤销和访问限制 | 用户主动转发链接后，访问控制取决于 token、密码和撤销状态 |
| AI 调用 | 服务端网关统一做额度、限流、缓存、审计、输出校验和兜底 | 第三方模型服务是外部处理边界，隐私说明必须清楚告知 |
| Nginx / 浏览器侧 | CSP、HSTS、X-Frame-Options、API no-store | CSP 不能为了兼容而放宽到危险策略 |

## 最关键的数据流

### 保存简历

```text
浏览器输入
  -> 前端净化
  -> 后端再次净化
  -> MySQL 保存草稿 / 版本
```

目的：即使攻击者绕过前端直接调 API，也不能把危险 HTML 直接落库并在预览、分享或 PDF 中执行。

### JD 诊断 / 岗位竞争力诊断 / 投递文案

```text
简历版本 + JD
  -> 后端组装上下文
  -> AI Gateway
  -> 第三方模型服务
  -> 结构化校验 / 风险兜底
  -> 数据库记录 / 前端展示
```

关键边界：

- JD 是岗位要求，不是候选人事实。
- 无证据要求只能输出“缺口 / 建议 / 保守表达”，不能写成候选人经历。
- 投递文案不能凭空生成公司名、工作经历或技术经验。
- AI 输出进入 HTML、数据库、PDF 前必须校验。

### PDF 导出

```text
后端创建导出任务
  -> 生成内部预览 URL
  -> Puppeteer 访问白名单页面
  -> 生成 PDF
  -> 用户下载
```

关键边界：

- 用户不能提交任意 URL 让服务器渲染。
- Puppeteer 不能访问本地文件、metadata IP、非白名单内网地址。
- PDF 任务异步执行，避免同步请求拖垮服务。
- 导出失败需要可追踪 taskId 和用户可理解的失败提示。

### 分享简历

```text
用户创建分享
  -> 后端生成分享快照
  -> 净化快照内容
  -> 生成长随机 token
  -> 前端展示短链接
  -> 持有链接者访问快照
```

说明：

- 短链接只是展示形式，不是自增 ID。
- 长 token 仍是访问控制核心。
- 用户应谨慎转发分享链接。
- 如果链接泄露，应立即撤销分享。

## AI 隐私边界

使用目标岗位分析、岗位竞争力诊断、投递文案等 AI 功能时，系统需要把必要的简历片段和 JD 文本发送给服务端模型网关，并可能转发给第三方模型服务处理。

因此建议：

- 不要输入身份证号、银行卡号、家庭住址、密钥、公司内部文档等敏感信息。
- JD 中如果包含公司内部保密信息，不要粘贴。
- AI 输出必须人工核对，不能直接当作事实。
- 对“不能硬写”的内容不要手动改成已完成经历。

## 用户如何降低泄露风险

- 只上传求职必要头像，不上传证件照原图或敏感照片。
- 分享链接只发给可信对象。
- 不在公开 Issue、评论、截图里暴露真实手机号、邮箱、简历全文、JD 原文。
- 导出 PDF 前检查个人信息是否适合对外发送。
- 发现异常登录、异常分享访问或错误数据，尽快反馈并撤销相关链接。
